MATCHING VARIABLES DISPLAYED TO USER:
====================================

✅ 6 Scoring Factors (0-100 each):
   1. AVAILABILITY (30%) - Energy capacity match
   2. PRICE (25%) - Affordability vs budget
   3. RELIABILITY (20%) - Seller rating + transactions
   4. DISTANCE (15%) - Geographic proximity
   5. RENEWABLE (5%) - Certification bonus
   6. TIMING (5%) - Availability window overlap

+ OVERALL SCORE (0-100) with recommendations:
   🟢 ≥75: Highly Recommended
   🟡 ≥50: Recommended  
   🔴 <50: Consider Alternatives

ARCHITECTURE:
=============

DEVICE LAYER:
   ESP32 Sensors → WiFi → MQTT Broker (Port 1883)

BROKER LAYER (Separate Container):
   Eclipse Mosquitto → Persistent Storage

BACKEND LAYER:
   Node.js Backend (3000) → PostgreSQL
   ↓
   ML Service (8002) → Calculations
   ↓
   Redis Cache (6379) → 5-min TTL

FRONTEND LAYER:
   React Native App → Shows 6 match variables

DATA FLOW:
1. ESP32 reads sensors (DHT22, ACS712, voltage divider) every 5 seconds
2. Publishes to MQTT: /solar/{device_id}/voltage, current, power, temperature, energy_today
3. Backend listener stores in device_readings table
4. When buyer searches, ML service queries recent device data
5. Calculates 6 scores for each seller
6. Returns weighted match score (0-100)
7. Frontend displays score breakdown to user
8. User selects matches → Creates smart_allocations

   FINAL SCORE = (50×0.30 + 90×0.25 + 60×0.20 + 25×0.15 + 50×0.05 + 70×0.05) = 59/100
   🟡 RECOMMENDED

WHY THIS IS BETTER THAN SIMPLE MATCHING:
=========================================

❌ Old way: "Find sellers within 5km" 
   Problem: Might miss better deals farther away

✅ New way: "Find best match considering price, reliability, AND distance"
   Solution: 93-score seller wins over 59-score even if both nearby

This algorithm ensures OPTIMAL allocations for both buyers and sellers,
maximizing energy trading efficiency while considering user preferences.

API ENDPOINTS:
===============

ML Service (Port 8002):
  POST /api/v1/match/find-sellers → Get ranked matches with 6 scores
  POST /api/v1/match/find-buyers → Reverse matching for sellers
  GET /api/v1/match/health → Service status

Backend (Port 3000):
  POST /api/v1/matching/find-sellers → With DB + caching
  GET /api/v1/matching/matches/:id → Detailed score breakdown
  POST /api/v1/matching/allocate → Create smart allocation
  GET /api/v1/matching/allocations/active → User's allocations
  DELETE /api/v1/matching/allocations/:id → Cancel allocation

MQTT (Port 1883):
  Publish: /solar/{device_id}/voltage, current, power, temperature, energy_today
  Subscribe: /command/{device_id}/shutdown, reset, config

DATABASE TABLES:
=================

smart_allocations
  - id, buyer_id, seller_id, allocated_kwh, price_per_kwh, match_score, status

device_readings
  - id, device_id, metric, value, timestamp

devices
  - id, device_id, user_id, status, last_heartbeat, total_kwh_today
